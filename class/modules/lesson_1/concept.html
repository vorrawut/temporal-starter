Introduction to Temporal

Lesson 1: Building the Foundation

by Vorrawut Judasri (Wut)

Why This Matters üí°

Before we dive in, think about this:

What happens when your backend job fails halfway? Who picks up the pieces?

This is where Temporal enters.

Objective üéØ

By the end of this lesson, you will:

  • ‚úÖ Understand what Temporal is and why it exists
  • ‚úÖ See how it solves painful issues in distributed systems
  • ‚úÖ Build the foundation to use it in real projects

üß† What is Temporal?

A workflow orchestration platform for reliable, scalable applications.

Think of it as an operating system for distributed systems.

It manages workflows, retries, failures, and state so you don't have to.

Temporal in Real Life üç≥

If your app is a restaurant, Temporal is the head chef:

  • üë®‚Äçüç≥ Makes sure orders are cooked correctly
  • üõ†Ô∏è Handles problems (like running out of ingredients)
  • üìã Tracks what's done and what's not
  • üîÑ Coordinates the entire kitchen workflow

üòñ What Problems Does It Solve?

Without Temporal:

  • üî• Scattered retry logic
  • üî• Manual error handling
  • üî• Hard to track long-running processes
  • üî• Lost messages & race conditions
  • üî• Difficult testing
  • üî• No visibility

Example: Without Temporal

fun processOrder(order: Order) {
    try {
        val payment = paymentService.charge(order.amount)
        val inventory = inventoryService.reserve(order.items)
        val shipping = shippingService.arrange(order)
    } catch (e: Exception) {
        // Uh-oh. Now what?
        // How do we recover?
        // What state are we in?
        // Should we retry everything?
    }
}

Problem: No coordination, no recovery, no visibility!

üí™ How Temporal Helps

With Temporal:

  • ‚úÖ Reliable execution (even after failure)
  • ‚úÖ Built-in state management
  • ‚úÖ Automatic retries
  • ‚úÖ Full visibility
  • ‚úÖ Easy local testing
  • ‚úÖ Workflow versioning

Example: With Temporal

@WorkflowInterface
interface OrderWorkflow {
    @WorkflowMethod
    fun processOrder(order: Order): OrderResult
}

class OrderWorkflowImpl : OrderWorkflow {
    override fun processOrder(order: Order): OrderResult {
        val payment = paymentActivity.charge(order.amount)
        val inventory = inventoryActivity.reserve(order.items)
        val shipping = shippingActivity.arrange(order)
        return OrderResult(payment, inventory, shipping)
    }
}

Result: Clean, reliable, observable workflow!

üß© Temporal Components

Workflows üéº

  • Define business logic (orchestration)
  • Deterministic & replayable
  • Control the flow of execution

Activities üé∏

  • Perform actual work (API, DB, etc.)
  • Can fail, retry, and run long
  • Handle non-deterministic operations

Components Continued

Workers üè≠

  • Execute workflows and activities
  • Scalable, fault-tolerant
  • Run your business logic

Temporal Server üè¢

  • Schedules, persists, coordinates everything
  • Manages state and execution
  • The brain of the operation

‚úÖ When to Use Temporal

Perfect for:

  • ‚úÖ Multi-step business processes
  • ‚úÖ Long-running jobs
  • ‚úÖ Processes across services
  • ‚úÖ Anything that needs reliability

Not ideal for:

  • ‚ùå Simple request/response APIs
  • ‚ùå Real-time (<ms latency) systems
  • ‚ùå Basic CRUD operations

‚öñÔ∏è Temporal vs Other Approaches

Approach Pros Cons
Manual Coordination Simple, flexible Hard to scale & maintain
Message Queues Async communication No orchestration built-in
State Machines Clear state tracking Poor error handling
Temporal Scalable, fault-tolerant Learning curve

üí° Best Practices

Mindset Shifts üß†

  • ‚úÖ Design in workflows, not service chains
  • ‚úÖ Keep non-determinism out of workflows
  • ‚úÖ Assume failures happen
  • ‚úÖ Start simple, evolve with use

Common Misconceptions üö´

Wrong Assumptions:

  • ‚ùå "It's just a message queue"
  • ‚ùå "Workflows are just functions"
  • ‚ùå "Too much complexity"
  • ‚ùå "Need to understand everything first"

Reality:

  • ‚úÖ It's a durable execution platform
  • ‚úÖ Start simple, learn by doing

Mental Model üß†

Temporal is your app's:

  • üß† Memory - Never forgets where you left off
  • üíæ State manager - Handles complex state transitions
  • üßë‚Äçüè´ Coordinator - Orchestrates distributed processes
  • üëÅÔ∏è Observer - Provides complete visibility
